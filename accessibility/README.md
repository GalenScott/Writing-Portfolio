# Accessibility Initiative & Presentation  
### Bringing Level Data Products Into Consistent WCAG 2.1 & 2.2 Compliance

This repository documents the work completed to research, design, and deliver a company‑wide Accessibility Presentation and testing framework. The initiative was created to increase organizational alignment around accessibility standards, improve product quality, and ensure all Level Data products meet or exceed WCAG 2.1 and 2.2 guidelines.

A11y: Accessibility 'shortcut' naming convention
---

## 📌 Purpose of the Initiative

The Problem: no consistency in how each Accessibility coding and testing was being done (if it was) for each product, and limited to no understanding of the time A11y testing takes, particularly for VPATS (Voluntary Product Accessibility Template) requested by Sales & Marketing for potential customers. Educating all departments needed to be done in a clear and efficient manner.

The primary goal of this project was to build shared understanding, consistency, and confidence across Product, Engineering, and QA teams regarding:

- Why accessibility matters  
- How accessibility testing works  
- What tools and processes are required  
- How to plan, execute, and document accessibility work  
- How to prepare for VPAT (Voluntary Product Accessibility Template) evaluations  

This work established a repeatable, scalable foundation for accessibility across the entire product suite.

## 🧩 What This Presentation Covers

### **1. Accessibility Tools Overview**
A curated set of tools used for manual and automated testing, including:

- Automated scanning tools  
- Screen readers  
- Keyboard‑only navigation  
- Touchscreen interaction testing  
- Element inspection tools  
- Contrast checking tools  

---
### **2. Testing Guides & Key Concepts**

The presentation breaks down how to test:

- Automated scans  
- Screen reader behavior  
- Keyboard‑only workflows  
- Touchscreen interactions  
- Page structure and markup using ANDI  
- Color contrast using WebAIM tools  

It emphasizes that **manual testing is essential**, because automated tools cannot interpret context, meaning, or workflow logic.

---

### **3. Preparation & Expectations for Accessibility Testing**

The presentation outlines what teams must provide before testing begins:

- Expected WCAG conformance level  
- Whether testing is for a VPAT or general regression  
- Product walkthroughs  
- Access to environments and workflows  
- Documentation of page states, modals, and generated files  

This ensures testing is accurate, efficient, and scoped correctly.

---

### **4. Estimating Scope of Work**

The deck explains how to determine realistic timelines by evaluating:

- Number of pages  
- Number of page states  
- Modal behavior  
- Component complexity  
- Third‑party components  
- Documentation outputs (PDF, CSV, etc.)

---

### **5. Planning the Work**

The presentation provides multiple testing strategies:

- **Option A:** Each tester focuses on one testing method across all pages  
- **Option B:** Each tester performs all testing types on assigned pages  
- **Option C:** Hybrid or team‑defined approach  

It also outlines pros/cons and emphasizes team comfort and efficiency.

---

### **6. What Accessibility Testing Involves**

The deck details the full testing expectations:

- Testing at 200% and 400% zoom  
- Screen reader testing  
- Keyboard‑only testing  
- Testing all page states  
- Testing generated documents  
- Testing across multiple browsers  

---

The presentation standardizes how to document and track accessibility issues:

- Spreadsheets  
- Guiding documents  
- WCAG Success Criteria references  
- W3C resources  
- Ticket creation and Epic structure  
- QA validation and UAT flow
